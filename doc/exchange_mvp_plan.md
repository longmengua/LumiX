# 交易所 MVP 專案落地規劃文件

> 目標：建立一個可真實營運、可讓真實用戶交易、可接外部做市商、可內部做市、可透過現貨、合約、槓桿產生收益的交易所 MVP。  
> 產品定位：第一版核心應包含「現貨 + U 本位永續合約 + 槓桿交易」，並同步具備做市商、風控、強平、對帳與後台營運能力。  
> 原則：不是做最多功能，而是把「可交易、可槓桿、可爆倉、可對帳、可風控、可做市、可營運」做完整。

---

## 1. 產品定位

### 1.1 產品組合邏輯

若交易所只提供單一交易產品，通常會遇到以下限制：

| 問題 | 影響 |
|---|---|
| 用戶吸引力不足 | 大部分高頻與高收益用戶更關心合約與槓桿 |
| 手續費收入低 | 純現貨成交頻率通常低於合約 |
| 做市商誘因不足 | 合約交易對流動性與交易量通常更有商業價值 |
| 差異化弱 | 純現貨很難和大型交易所競爭 |
| 留存不足 | 沒有槓桿、資金費率、倉位管理，交易體驗偏弱 |

因此，第一版 MVP 建議包含：

```text
現貨交易
  +
U 本位永續合約
  +
槓桿交易
  +
外部做市商
  +
內部做市商
  +
完整風控與對帳
```

## 2. MVP 範圍

### 2.1 第一版必要產品

| 產品 | 是否納入 MVP | 說明 |
|---|---:|---|
| 現貨交易 | 必要 | 作為資產兌換、價格基礎與用戶入門 |
| U 本位永續合約 | 必要 | 核心交易收入來源，支援高槓桿交易 |
| 槓桿交易 | 必要 | 支援借幣、保證金、利息與強平 |
| Open API | 必要 | 給外部做市商與高階交易者 |
| 外部做市商 | 必要 | 解決深度、spread、流動性 |
| 內部做市商 | 必要 | 早期補流動性與價格跟隨 |
| 後台營運 | 必要 | 查帳、審核、風控、事故處理 |
| 對帳系統 | 必要 | 資產、成交、倉位、借貸、手續費都要可對 |
| 風控系統 | 必要 | 合約與槓桿沒有風控不能上線 |

### 2.2 第一版交易對建議

| 類型 | 第一批交易對 |
|---|---|
| 現貨 | BTC/USDT、ETH/USDT、SOL/USDT |
| U 本位永續 | BTCUSDT Perp、ETHUSDT Perp、SOLUSDT Perp |
| 槓桿現貨 | BTC/USDT、ETH/USDT |

第一版不要一開始開太多小幣。  
合約與槓桿的小幣風險很高，容易插針、爆倉、被操縱。

## 3. 第一版仍然不建議做的功能

| 功能 | 是否放進 V1 | 原因 |
|---|---:|---|
| 幣本位合約 | 暫不放 | 帳務與保證金複雜度更高 |
| 期權 | 暫不放 | 定價、希臘值、風控複雜 |
| 組合保證金 | 暫不放 | 對風控模型要求高 |
| 跟單 | 暫不放 | 需要交易員體系與分潤 |
| 理財 | 暫不放 | 涉及資產池與收益來源 |
| Launchpad | 暫不放 | 非交易核心 |
| P2P | 暫不放 | 法幣、商家、申訴複雜 |
| 複雜 VIP | 暫不放 | 可以先做簡單費率等級 |
| 全幣種全鏈 | 暫不放 | 錢包與風控成本太高 |

## 4. 修正版總體開發順序

### 4.1 推薦順序

```text
統一資產帳本設計
  ↓
帳號 / 權限 / 後台骨架
  ↓
充值 / 提現 / 錢包 Gateway
  ↓
行情基礎 / 外部價格源 / 指數價格
  ↓
共用撮合引擎
  ↓
現貨交易核心
  ↓
U 本位永續合約核心
  ↓
合約風控 / 保證金 / 強平
  ↓
槓桿交易 / 借貸 / 利息 / 強平
  ↓
Open API
  ↓
外部做市商
  ↓
內部做市商
  ↓
後台營運 / 風控 / 對帳
  ↓
小流量上線
```

### 4.2 為什麼要先做統一帳本

因為現貨、合約、槓桿都會動到資產。如果帳本一開始只為現貨設計，後面補合約與槓桿會大返工。

第一版帳本就要支援：

| 帳戶 | 用途 |
|---|---|
| 現貨帳戶 | 充值、提現、現貨交易 |
| 合約帳戶 | 永續合約保證金、盈虧、資金費率 |
| 槓桿帳戶 | 借幣、還幣、利息、保證金 |
| 做市商帳戶 | 外部 / 內部做市資產與權限 |
| 平台收入帳戶 | 手續費、提幣費、利息收入 |
| 保險基金帳戶 | 合約穿倉與強平風險承接 |

## 7. 統一資產帳本設計

### 7.1 帳戶分類

| 帳戶 | 資產用途 |
|---|---|
| 現貨帳戶 | 充值、提現、現貨交易 |
| 合約帳戶 | 永續合約保證金、已實現盈虧、未實現盈虧、資金費率 |
| 槓桿帳戶 | 借入資產、負債、利息、槓桿交易 |
| 平台收入帳戶 | 交易手續費、提幣手續費、利息收入 |
| 保險基金帳戶 | 合約穿倉、強平虧損、極端行情處理 |
| 做市商帳戶 | 外部與內部做市資產隔離管理 |

### 7.2 帳本必須支援的資產動作

| 動作 | 說明 |
|---|---|
| 入帳 | 充值、轉入、盈虧入帳 |
| 扣帳 | 提現、手續費、虧損扣除 |
| 凍結 | 下單、提現、保證金占用 |
| 解凍 | 撤單、提現失敗、保證金釋放 |
| 轉帳 | 現貨帳戶、合約帳戶、槓桿帳戶之間劃轉 |
| 借入 | 槓桿借幣 |
| 還款 | 歸還本金與利息 |
| 利息 | 槓桿利息計算 |
| 資金費率 | 合約多空資金費交換 |
| 強平扣帳 | 強平時扣除保證金與虧損 |
| 保險基金入帳 | 強平剩餘資產或穿倉處理 |

### 7.3 資產安全底線

| 規則 | 說明 |
|---|---|
| 所有資產變動都必須有流水 | 不允許直接改餘額 |
| 可用與凍結必須分離 | 下單、提現、保證金都要凍結 |
| 每個帳戶獨立記帳 | 現貨、合約、槓桿不能混亂 |
| 不允許負數 | 除了明確定義的槓桿負債欄位 |
| 所有結算必須冪等 | 避免重複扣款或重複入帳 |
| 每日必須對帳 | 對資產、成交、倉位、負債、費用 |
| 後台調帳必須審批 | 不允許單人直接改資產 |
| 保險基金獨立帳戶 | 不可和平台收入混用 |

## 9. U 本位永續合約核心

### 9.1 為什麼 MVP 必須有 U 本位永續

| 原因 | 說明 |
|---|---|
| 交易量核心 | 合約通常是交易所交易量與收入核心 |
| 用戶需求強 | 高階用戶主要交易槓桿與永續 |
| 做市商更願意接 | 合約有更高流動性與交易頻率 |
| 平台收益更明確 | 手續費、資金費率相關收益、清算流程都可設計 |
| 競爭力更強 | 純現貨交易所很難吸引用戶 |

### 9.2 第一版合約範圍

| 項目 | MVP 設計 |
|---|---|
| 合約類型 | U 本位永續 |
| 保證金幣種 | USDT |
| 第一批合約 | BTCUSDT、ETHUSDT、SOLUSDT |
| 倉位模式 | 單向持倉優先，雙向持倉可第二版 |
| 保證金模式 | 逐倉優先，交叉可第二版或同步設計 |
| 槓桿倍數 | BTC / ETH 可較高，SOL 較低 |
| 強平 | 必須有 |
| 資金費率 | 必須有 |
| 標記價格 | 必須有 |
| 指數價格 | 必須有 |
| 保險基金 | 必須有 |
| ADL 自動減倉 | 可簡化，但必須有極端情況處理策略 |

### 9.3 合約必要概念

| 概念 | 說明 |
|---|---|
| 合約帳戶 | 用戶放入合約保證金的帳戶 |
| 倉位 | 用戶多單或空單 |
| 開倉 | 建立新倉位 |
| 平倉 | 減少或關閉倉位 |
| 保證金 | 支撐倉位的資產 |
| 初始保證金 | 開倉需要的最低保證金 |
| 維持保證金 | 不被強平的最低要求 |
| 未實現盈虧 | 根據標記價格計算的浮動盈虧 |
| 已實現盈虧 | 平倉後落地的盈虧 |
| 標記價格 | 用於盈虧與強平，不直接用最新成交價 |
| 指數價格 | 來自外部多個交易所的參考價格 |
| 資金費率 | 多空之間定期支付 |
| 強平價格 | 倉位會被強制平倉的價格 |
| 保險基金 | 處理穿倉與強平損失 |

### 9.4 合約下單流程

```text
用戶 / 做市商提交合約訂單
  ↓
檢查合約交易對、價格、數量、精度
  ↓
檢查槓桿與風險限額
  ↓
計算初始保證金
  ↓
檢查合約帳戶可用保證金
  ↓
凍結保證金
  ↓
建立合約訂單
  ↓
進入撮合引擎
  ↓
成交後更新倉位
  ↓
更新保證金占用
  ↓
計算手續費
  ↓
推送訂單、成交、倉位更新
```

### 9.5 合約倉位流程

```text
成交事件
  ↓
查詢用戶當前倉位
  ↓
判斷是開倉、加倉、減倉、平倉、反向開倉
  ↓
更新倉位數量與均價
  ↓
計算已實現盈虧
  ↓
計算未實現盈虧
  ↓
更新保證金占用
  ↓
更新強平價格
  ↓
寫入倉位流水
```

### 9.6 資金費率流程

```text
每個資金費率週期
  ↓
計算每個合約的資金費率
  ↓
找出所有持倉用戶
  ↓
多空雙方計算應收 / 應付
  ↓
從應付方合約帳戶扣款
  ↓
給應收方合約帳戶入帳
  ↓
寫入資金費率流水
  ↓
納入每日對帳
```

### 9.7 標記價格與指數價格流程

```text
外部價格源：
Binance / OKX / Bybit / Coinbase 等
  ↓
價格清洗：
剔除異常價格、過期價格、偏離過大價格
  ↓
計算指數價格
  ↓
結合資金費率基差
  ↓
生成標記價格
  ↓
用於：
未實現盈虧、保證金率、強平判斷
```

## 10. 合約強平系統

### 10.1 強平為什麼是第一版必做

合約沒有強平系統不能上線。  
因為只要用戶虧損超過保證金，平台就可能承擔損失。

### 10.2 強平判斷流程

```text
標記價格更新
  ↓
重新計算每個倉位未實現盈虧
  ↓
計算帳戶權益
  ↓
計算維持保證金需求
  ↓
判斷保證金率
  ↓
如果低於安全線
  ↓
進入強平流程
```

### 10.3 強平流程

```text
倉位觸發強平
  ↓
凍結該倉位與相關委託
  ↓
取消用戶該合約未成交委託
  ↓
將倉位交給強平引擎
  ↓
強平引擎用合理價格平倉
  ↓
若平倉後有剩餘資產
    ↓
    部分進保險基金或返還用戶
  ↓
若出現穿倉
    ↓
    使用保險基金承接
  ↓
保險基金不足
    ↓
    啟動極端風險處理
```

### 10.4 強平系統必備能力

| 能力 | 說明 |
|---|---|
| 實時計算保證金率 | 標記價格變動後即時更新 |
| 取消未成交委託 | 強平前先釋放保證金 |
| 部分強平 | 優先降低風險，不一定一次全平 |
| 強平訂單管理 | 防止強平單打穿市場 |
| 保險基金 | 承接穿倉風險 |
| 風險限額 | 不同倉位大小對應不同保證金要求 |
| 後台監控 | 查看即將強平、已強平、穿倉紀錄 |
| Kill Switch | 極端行情可暫停合約開倉或交易 |

### 10.5 合約風險限額

| 項目 | 說明 |
|---|---|
| 最大槓桿 | 不同交易對不同上限 |
| 最大單筆下單量 | 防止大單打穿 order book |
| 最大持倉量 | 防止單一用戶風險過大 |
| 階梯維持保證金 | 倉位越大，維持保證金越高 |
| 價格偏離保護 | 下單價格不能偏離標記價格太遠 |
| 市價單保護 | 市價單需要最大滑點限制 |
| 強平保護 | 強平單不能無限制打穿市場 |

## 11. 槓桿交易核心

### 11.1 MVP 槓桿範圍

| 項目 | MVP 設計 |
|---|---|
| 槓桿模式 | 現貨槓桿 |
| 第一版交易對 | BTC/USDT、ETH/USDT |
| 借貸資產 | USDT、BTC、ETH |
| 保證金模式 | 可先做逐倉槓桿 |
| 利息 | 小時計息或日計息 |
| 強平 | 必須有 |
| 風險率 | 必須有 |
| 自動還款 | 可第二版 |
| 自動借款 | 可第二版 |

### 11.2 槓桿帳戶概念

| 概念 | 說明 |
|---|---|
| 槓桿資產 | 用戶自有資產 |
| 借入資產 | 用戶從平台借的資產 |
| 負債 | 借入本金 + 利息 |
| 淨資產 | 資產 - 負債 |
| 風險率 | 判斷是否需要強平 |
| 利息 | 使用借入資產需要支付 |
| 還款 | 歸還借入本金與利息 |

### 11.3 槓桿借幣流程

```text

  ↓

  ↓
計算可借額度
  ↓
檢查平台可借資產池
  ↓
借幣成功
  ↓
增加用戶槓桿帳戶資產
  ↓
增加用戶負債
  ↓
開始計息
```

### 11.4 槓桿交易流程

```text
用戶在槓桿帳戶下單
  ↓
檢查槓桿帳戶資產與負債
  ↓
檢查風險率
  ↓
凍結交易資產
  ↓
進入現貨撮合引擎
  ↓
成交後更新槓桿帳戶資產
  ↓
重新計算負債與風險率
```

### 11.5 槓桿還款流程

```text
用戶申請還款
  ↓
計算本金與利息
  ↓
檢查可用資產是否足夠
  ↓
扣除還款資產
  ↓
降低負債
  ↓
寫入還款流水
  ↓
重新計算風險率
```

### 11.6 槓桿強平流程

```text
價格變動
  ↓
重新估算槓桿帳戶資產價值
  ↓
計算負債與利息
  ↓
計算風險率
  ↓
低於強平線
  ↓
取消未成交委託
  ↓
強制賣出資產或買回負債資產
  ↓
償還負債
  ↓
剩餘資產返還用戶
  ↓
不足部分進入壞帳 / 保險基金處理
```

## 13. 內部做市商

### 13.1 內部做市商原則

| 原則 | 說明 |
|---|---|
| 必須走 Open API | 不得直接改 order book |
| 必須走帳本 | 不得繞過資產流水 |
| 必須走撮合 | 不得特殊成交 |
| 必須有風控 | 防止無限虧損 |
| 必須有 kill switch | 可一鍵停止 |
| 必須和用戶公平撮合 | 不得傷害真實用戶 |

### 13.2 內部做市範圍

| 類型 | 是否做市 |
|---|---:|
| 現貨 | 是 |
| U 本位永續 | 是，優先 |
| 槓桿 | 透過現貨 order book 間接支持 |

### 13.3 內部做市流程

```text
外部價格源
  ↓
計算參考價格
  ↓
讀取自家 order book
  ↓
讀取自身庫存與倉位
  ↓
計算 bid / ask 報價
  ↓
檢查最大庫存 / 最大倉位 / 最大虧損
  ↓
透過 Open API 掛單
  ↓
監控成交
  ↓
調整報價與庫存
  ↓
異常時撤單或停止
```

### 13.4 合約內部做市風控

| 風控項 | 說明 |
|---|---|
| 最大倉位 | 防止單邊暴露 |
| 最大槓桿 | 內部做市不應高槓桿失控 |
| 最大日虧損 | 達到即停止 |
| 最大掛單量 | 防止打穿市場 |
| 標記價格偏離 | 偏離外部價格時撤單 |
| 波動率保護 | 極端行情停止做市 |
| 強平保護 | 內部 MM 不應被強平造成系統風險 |
| 手動 kill switch | 後台可停止所有內部 MM |

### 18.7 帳戶劃轉

| 劃轉方向 | MVP 是否需要 |
|---|---:|
| 現貨帳戶 → 合約帳戶 | 必要 |
| 合約帳戶 → 現貨帳戶 | 必要 |
| 現貨帳戶 → 槓桿帳戶 | 必要 |
| 槓桿帳戶 → 現貨帳戶 | 必要 |

| 槓桿帳戶 → 合約帳戶 | 可第二階段 |

劃轉流程：

```text
用戶選擇來源帳戶與目標帳戶
  ↓
輸入資產與數量
  ↓
檢查來源帳戶可用餘額
  ↓
檢查目標帳戶是否允許該資產
  ↓
扣來源帳戶可用餘額
  ↓
增加目標帳戶可用餘額
  ↓
寫入帳戶劃轉流水
  ↓
前台更新資產
```

## 26. 最重要結論

產品落地判斷：

```text
單一現貨產品不足以支撐完整交易所營運。
合約是交易所 MVP 的核心。
槓桿也是必要產品。
但合約與槓桿不能只是功能頁面，必須同時具備：
保證金系統
強平系統
指數價格
標記價格
資金費率
保險基金
風控後台
每日對帳
做市商 API
```

最終落地順序應該是：

```text
統一帳本
  ↓
錢包與資產
  ↓
現貨基礎
  ↓
合約核心
  ↓
合約強平
  ↓
槓桿借貸
  ↓
Open API
  ↓
做市商
  ↓
後台風控
  ↓
對帳上線
```

若要讓真實用戶使用、讓做市商願意接入、讓平台具備交易收入能力，第一版應按「現貨 + 合約 + 槓桿」完整設計。
